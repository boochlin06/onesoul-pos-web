# OneSoul POS 系統介紹

OneSoul POS 是一套專為零售門市量身打造的輕量化、高效率銷售點管理系統 (Point of Sale)。本系統獨創運用 **Google Sheets** 作為雲端資料庫（透過 Google Apps Script 提供 API 服務），並搭配現代化的 **React Web 介面**，提供店鋪人員極致流暢的操作體驗。

## 🌟 核心理念
系統旨在解決傳統 POS 系統過於笨重、維護成本高昂的問題。透過無伺服器 (Serverless) 的 Google 雲端生態系，結合極速的網頁前端技術，帶來：
- **零月租費**的資料庫架構（Google Sheets）。
- **即時同步**的雲端試算表，方便管理者隨時查看報表。
- **跨平台**支持，只要有瀏覽器（平板、電腦）即可快速結帳。

## 🚀 系統架構
1. **Frontend (前端 Web 應用)**
   - 採用 **React 19 + Vite** 建置的單頁應用程式 (SPA)。
   - 使用 **Tailwind CSS** 打造現代化且直覺的介面設計。
   - **Domain-Driven Hooks 架構**：業務邏輯按領域拆分為獨立 hooks（`useBanner`、`useMembers`、`usePrizes`、`useSales`、`useInventory`、`useCheckout`），遵循 SOLID 單一職責原則。
   - **純函數業務邏輯層** (`logic/checkout.ts`)：結帳計算、驗證、自動帶入等核心邏輯提取為 framework-agnostic 純函數，100% 可測試。
   - **Vitest 單元測試**：157+ 個測試涵蓋金額計算、防呆驗證、自動帶入、UI 元件等關鍵路徑（覆蓋率 80%）。
   - **Playwright E2E scaffold**：客戶面頁面端對端測試。
   - **環境變數管理**：GAS URL 與 API Key 透過 `.env` + Vite `import.meta.env` 管理，不暴露於原始碼。
2. **Backend (後端 GAS API)**
   - 採用 **Google Apps Script (GAS)** 開發。
   - 負責接收前端請求並讀寫指定的 Google Sheets 工作表。
   - 處理核心業務邏輯：結帳扣庫存、點數更新、作廢訂單同步、資料撈取。
   - **API Key 驗證**：`doPost` 入口比對 `PropertiesService` 存放的 API Key，阻擋未授權請求。
   - **LockService 並發控制**：結帳、關帳、作廢三大寫入操作使用 `getScriptLock().waitLock(30000)` 互斥鎖，防止多門市同時操作導致資料覆蓋。
   - **CacheService 效能快取**：會員消費紀錄查詢結果快取 5 分鐘，降低 Google Sheets API 呼叫頻率。

## ✨ 主要功能亮點
- **極致流暢結帳與購物車**：支援快速掃描、搜尋商品並加入購物車，自動計算折扣與應付金額。
- **高風險結帳攔截 (Checkout Confirmation Modal)**：送出前強制二次確認機制，將明細重組成三大發放指標（帶走、換點、贈送）。並搭載智慧防呆：大獎轉點數強制鎖定功能 與 小獎帶走實體提示，防堵人為結帳失誤。
- **庫存盤點系統 (Inventory Check)**：專為實體門市設計的手機卡片盲點版面，內建進度條、類別色區與逐筆備註功能；盤點完成後交由管理者審核並寫入實際庫存，防止誤觸。
- **點數套結帳**：獎項庫 J 欄標記「點數」的套組自動偵測，金額欸零，改為輸入「扣抵點數」（預設 = 單抽價/20 × 抽數），K 欄寫入淨點數。
- **抽數防呆**：輸入抽數不可超過該獎項在套裡的總數量。
- **強大會員與點數系統**：即時查詢會員、發放點數與折抵消費。
- **交易群組化顯示**：各項商品明細會根據交易 ID (checkoutUID) 自動群組為單筆訂單，版面清晰。
- **作廢訂單與點數返還**：在「當日銷售」中一鍵作廢訂單，系統自動比對並退回/扣除會員點數，保證帳務準確。
- **多門市支援**：內建門市切換功能（如：竹北、金山），確保各門市資料獨立與過濾正確。
- **財務數據看板**：當日銷售頁面直接統整今日總營收、現金、刷卡、匯款與點數發放狀況。
- **9 道結帳防呆驗證**：空單、缺電話、未查詢會員、負數金額、實收金額不符、點數不足等全面攔截。

## 🔒 安全性
- **Google OAuth 登入閘門**：POS 端所有操作需先完成 Google Sign-In，後端 JWT 解碼驗證 `iss`（issuer）+ `exp`（過期時間）+ `aud`（audience），僅接受 `accounts.google.com` 簽發的 token。
- **email 白名單**：`ALLOWED_EMAILS` 嚴格限制可操作帳號。
- **API Key 認證**：所有前端請求必須攜帶有效 API Key，GAS 端於 `doPost` 入口即時驗證。
- **速率限制**：客戶登入 API 使用 `CacheService` 實作同一電話 60 秒內最多 5 次嘗試，防止暴力破解。
- **原子化點數操作**：`addMemberPointsByPhone` 在 `ScriptLock` 內讀取即時 DB 值進行相對加減，防止 race condition。
- **deleteRow 排序保護**：批量刪除時顯式 `sort(b-a)` 確保行號從大到小刪除，防止偏移。
- **環境變數隔離**：敏感資訊（URL、Key）不進版控，透過 `.env` 管理。
- **LockService 互斥鎖**：寫入操作加入 Script-level 互斥鎖，最長等待 30 秒自動 timeout 通知使用者。

## 🚢 部署流程
- **前端**：`dev` 分支開發 → 合併到 `main` → GitHub Actions 自動建置並部署到 GitHub Pages。
- **後端**：使用 `npm run push:dev` 或 `npm run push:prod` 一鍵推送至對應之 GAS 環境並發布新版。
- **分支策略**：`dev`（開發/測試）→ `main`（生產部署），GitHub Pages 環境保護限定 `main` 才能部署。
- **Wiki 自動同步**：`docs/` 或 `wiki/` 變更推送到 `main` 後，GitHub Action 自動同步至 Wiki。
