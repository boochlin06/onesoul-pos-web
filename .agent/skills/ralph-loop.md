# Ralph Loop

**R**ead → **A**ct → **L**oop → **P**rint → **H**alt

遇到需要反覆嘗試、觀察結果、調整的任務時，走這個迭代循環，直到成功或明確失敗為止。

適用場景：Debug、測試修復、配置調整、整合測試。

---

## 循環步驟

```
┌─────────────────────────────────────────┐
│  R — READ                               │
│  讀取當前狀態：log、錯誤訊息、輸出結果     │
├─────────────────────────────────────────┤
│  A — ACT                                │
│  根據觀察執行一個最小可行的修正            │
├─────────────────────────────────────────┤
│  L — LOOP                               │
│  跑測試 / 啟動服務 / 觸發操作，觀察結果    │
├─────────────────────────────────────────┤
│  P — PRINT                              │
│  輸出這輪的結果：成功 ✅ 或 失敗 ❌ + 原因  │
├─────────────────────────────────────────┤
│  H — HALT                               │
│  成功 → 停止並回報。失敗超過 N 輪 → 停止   │
│  並說明目前卡在哪裡                       │
└─────────────────────────────────────────┘
```

---

## 規則

### 每輪只做一件事
- 不要一次改 3 個地方再測試，找不到是哪個改動有效
- 最小修改原則：改最少的程式碼，看是否解決問題

### 狀態要記錄
每輪結束時在腦中（或 task.md 裡）記錄：
```
Loop 1: 假設是 X → 測試 → 結果：Y → 排除 X，懷疑 Z
Loop 2: 假設是 Z → 改了 A → 測試 → 成功 ✅
```

### 停止條件
- ✅ 成功：問題解決，回報結論
- ❌ 超過 5 輪仍失敗：停下來，說明：
  - 目前確認的事實
  - 目前排除的假設
  - 剩餘的可能原因
  - 需要使用者提供什麼資訊

### 不做無用的迴圈
如果兩輪連續輸出完全相同的錯誤，表示做法方向錯了，不要繼續硬幹。

## 本專案常用的 Loop 指令

```bash
# 跑前端單元測試
cd web && npm run test -- --run

# 跑特定測試檔
cd web && npm run test -- --run src/logic/checkout.test.ts

# 啟動 dev server（背景）
cd web && npm run dev

# 推 GAS 到 dev 環境
npm run push:dev

# 查 GAS 用量
# → GAS 編輯器跑 checkQuotaUsage()
```

## Debug 常見起點（本專案）

| 症狀 | 先看哪裡 |
|------|---------|
| 結帳 API 失敗 | GAS 執行紀錄 → `apiCheckout` 的 catch 訊息 |
| 點數錯誤 | `_addPointsUnsafe` 的 return value（-1/-2） |
| LINE 通知沒發到 | 確認 `LINE_CHANNEL_ACCESS_TOKEN` + `API設定` Sheet |
| 前端呼叫 API 失敗 | 瀏覽器 Network → 看 doPost response body |
| 關帳資料不對 | 確認 `uidSet` 去重邏輯、欄位索引 [14] = checkoutUID |
| 當日銷售有重複 | 確認 checkoutUID grouping 邏輯、void 標記 [22] |
